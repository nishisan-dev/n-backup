# Major Release v4.0.0 — Production Readiness

## Contexto

O [relatório de production readiness](file:///home/lucas/Projects/n-backup/planning/production_readiness_2026-03-23.md) de 23/03/2026 identificou 8 gaps para GA, após incidente real de checksum mismatch em produção (sessão `45d0567a`). O principal gap (CRC32 por chunk) altera o layout do `ChunkHeader` no wire protocol — **breaking change** que justifica o bump para **v4.0.0** (Protocol v6).

Versão atual: **v3.4.0** (Protocol v5).

> [!IMPORTANT]
> **API Breaking Change**: O `ChunkHeader` muda de 9B para 13B com a adição de CRC32. Clients v3.x **não são compatíveis** com servers v4.x e vice-versa. Essa é uma decisão consciente do usuário — não haverá fallback para protocolo legado.

---

## Faseamento

A implementação será realizada em **3 fases**, ordenadas por prioridade e dependência:

| Fase | Itens | Prioridade | Commits esperados |
|------|-------|-----------|-------------------|
| **Fase 1** — Integridade de Dados | CRC32 por chunk + fsync default | P0 | 2-3 |
| **Fase 2** — Gestão de Lifecycle | MaxBackupDuration per-retry + AssemblyProgress + hash parcial + otimização lazy assembler | P1 | 4-5 |
| **Fase 3** — Testes e Documentação | Integração + specification | P2 | 2-3 |

## Fase 1 — Integridade de Dados (P0)

### 1.1 CRC32 por chunk no ChunkHeader (Protocol v6)

O `ChunkHeader` atual é:
```
[GlobalSeq uint32 4B] [Length uint32 4B] [SlotID uint8 1B] = 9 bytes
```

O novo `ChunkHeader` será:
```
[GlobalSeq uint32 4B] [Length uint32 4B] [SlotID uint8 1B] [CRC32 uint32 4B] = 13 bytes
```

O CRC32 é calculado sobre os **dados do chunk** (payload), não sobre o header. Usa IEEE polynomial (CRC-32, hardware-accelerated via `hash/crc32.MakeTable(crc32.IEEE)`).

#### [MODIFY] [frames.go](file:///home/lucas/Projects/n-backup/internal/protocol/frames.go)
- Bump `ProtocolVersion` de `0x05` para `0x06`
- `ChunkHeaderSize` de `9` para `13`
- Adicionar campo `CRC32 uint32` na struct `ChunkHeader`
- Adicionar erro `ErrChunkCRCMismatch`

#### [MODIFY] [writer.go](file:///home/lucas/Projects/n-backup/internal/protocol/writer.go)
- `WriteChunkHeader()`: aceitar parâmetro `crc32 uint32`, serializar 4B extras (big-endian) após SlotID

#### [MODIFY] [reader.go](file:///home/lucas/Projects/n-backup/internal/protocol/reader.go)
- `ReadChunkHeader()`: ler 4B extras de CRC32, popular `ChunkHeader.CRC32`

#### [MODIFY] [dispatcher.go](file:///home/lucas/Projects/n-backup/internal/agent/dispatcher.go)
- `emitChunk()` (L247-310): calcular `crc32.ChecksumIEEE(data)` antes de emitir, incluir no header de 13B
- `readChunkFrame()` (L655-685): ajustar leitura para novo `ChunkHeaderSize`
- `syncStreamAfterReconnect()` (L513-573): ajustar validação de alinhamento para 13B

#### [MODIFY] [handler_parallel.go](file:///home/lucas/Projects/n-backup/internal/server/handler_parallel.go)
- `receiveParallelStream()` (L429-555): após ler header e payload, validar CRC32 do payload. Se mismatch → rejeitar chunk com log `chunk_crc_mismatch` e retornar erro (força reconexão do stream)
- Importar `hash/crc32`

#### [MODIFY] [assembler.go](file:///home/lucas/Projects/n-backup/internal/server/assembler.go)
- `finalizeLazy()` (L615-647): revalidar CRC32 antes de copiar para output se o CRC foi armazenado junto ao chunk em staging
- Adicionar campo `CRC32` à struct `pendingChunk` para persistir o CRC recebido
- Em `writeChunkLazy()`: armazenar CRC no `pendingChunk` para validação posterior

## Fase 2 — Gestão de Lifecycle (P1)

### 2.1 MaxBackupDuration independente por retry

Atualmente, `scheduler.go:153` aplica `context.WithTimeout(MaxBackupDuration)` ao **job todo**, e `backup.go:580` aplica outro `WithTimeout` ao `WaitAllSenders`. Retries herdam o contexto do job — se o job consumiu 23h, o retry tem apenas 1h.

#### [MODIFY] [scheduler.go](file:///home/lucas/Projects/n-backup/internal/agent/scheduler.go)
- Remover o `context.WithTimeout(context.Background(), MaxBackupDuration)` do `executeJob()` (L153)
- Substituir por contexto sem timeout (ou timeout de safety net muito longo, ex: 72h)
- O timeout real será aplicado dentro do `runFn` — por tentativa individual

#### [MODIFY] [backup.go](file:///home/lucas/Projects/n-backup/internal/agent/backup.go)
- No `RunBackup()`: envolver cada iteração do loop de retry (L141+) com `context.WithTimeout(ctx, MaxBackupDuration)`
- No `runParallelBackup()`: o `sendersCtx` (L580) já tem timeout, mas precisa ser criado fresh em cada invocação
